# SPR 系统升级路线图

> 目标：把当前可用但仍偏个人化、补丁化的系统，逐步升级为库存可靠、可追溯、可恢复、可测试、可维护、可安全交付的业务系统。
>
> 本文档是长期执行清单。后续每次改动都应先更新本文档中的状态，再实施代码或数据库变更。

## 一、当前项目基线

### 当前技术形态

- 前端以静态 HTML、CSS、JavaScript 为主。
- 业务逻辑主要集中在 `store-app.js`、`store-after-sales.js` 和若干补丁文件。
- 数据库使用 Supabase PostgreSQL。
- 订单、库存、经销商出库导入等逻辑部分在前端，部分在数据库函数和触发器中。

### 当前关键业务表

- `employees`
- `products`
- `employee_store_assets`
- `dealer_employee_mappings`
- `raw_dealer_outbounds`
- `van_stocks`
- `sales_orders`
- `sales_order_items`

### 当前库存起算规则

- `2026-07-01 00:00:00 +08` 以前的订单：只保留查看，不参与库存。
- `2026-07-01 00:00:00 +08` 及以后的订单：参与库存扣减、修改和删除恢复。

### 已完成的临时保护

- 已备份订单主表和订单明细。
- 已对员工 `S260401018` 做过单员工库存清理和 7 月后订单回放测试。
- 已增加 7 月库存分界补丁。

# 总体实施顺序

按以下顺序推进，不同时大规模并行：

1. 库存统一与库存流水
2. 核心自动化测试
3. 备份、恢复与审计
4. 登录、权限与 RLS
5. 业务模块拆分

每个阶段都必须满足：

- 先备份
- 小范围测试
- 核对数据
- 有回滚方式
- 验收通过后再扩大范围

# 第一阶段：库存统一与库存流水

## 目标

所有库存变化统一从数据库完成，前端不再直接计算最终库存并覆盖 `van_stocks`。

最终应做到：

- 当前库存准确
- 每次变化可追溯
- 可以解释库存为何是当前数字
- 可以按员工、商品、订单、导入批次重算
- 重复请求不会重复扣减或恢复

## 1.1 梳理所有库存入口

### 要检查的入口

- 经销商出库导入
- 库存管理页覆盖导入
- 普通开单
- 整件开单
- 散数开单
- 拼盒开单
- 售后收回
- 修改订单
- 删除订单
- 人工调整库存
- 清空并重新建立库存

### 输出

形成一份库存事件表，明确每种事件：

- 增加还是减少库存
- 使用哪个日期
- 是否允许负数
- 是否可撤销
- 是否可重复执行
- 对应来源表和来源编号

### 完成标准

- [ ] 所有库存入口均已列出
- [ ] 每个入口的库存方向已确认
- [ ] 7 月前后规则已确认
- [ ] 售后库存规则已确认
- [ ] 拼盒换算规则已确认

- `DEALER_INBOUND`
- `SALE_OUTBOUND`

- `ORDER_EDIT_ADJUSTMENT`
- `ORDER_DELETE_RESTORE`
- `MANUAL_ADJUSTMENT`
- `RESET`

### 关键约束

- 同一个来源动作只能写入一次。
- 订单提交、修改、删除必须具备幂等性。
- 流水写入和 `van_stocks` 更新必须在同一个事务内完成。

### 完成标准

- [ ] 表结构建立
- [ ] 索引建立
- [ ] 来源唯一约束建立
- [ ] 测试插入和查询通过

## 1.3 建立统一库存数据库函数

建议逐步建立以下 RPC：

- `apply_stock_movement(...)`
- `submit_sales_order_v3(...)`
- `update_sales_order_v2(...)`
- `delete_sales_order_v2(...)`
- `import_opening_stock(...)`
- `import_current_stock_snapshot(...)`
- `adjust_stock(...)`
- `rebuild_employee_stock(...)`

### 规则

- 前端只传业务输入，不传最终库存值。
- 数据库负责读取当前库存、计算变化、写流水、更新当前库存。
- 所有函数必须返回明确结果：成功、失败、影响条数、库存前后值。

### 完成标准

- [ ] 普通开单迁移到新 RPC
- [ ] 售后迁移到新 RPC
- [ ] 拼盒迁移到新 RPC
- [ ] 改单迁移到新 RPC
- [ ] 删单迁移到新 RPC
- [ ] 前端不再直接 upsert 最终库存

## 1.4 明确库存导入语义

库存管理页需要区分两种导入：

### 期初库存导入

含义：某个起算日的库存基数。

公式：

```text
当前库存 = 期初库存 + 起算日后入库 - 起算日后销售 + 售后及调整
```

### 当日盘点覆盖

含义：导入当天真实盘点结果。

规则：

- 不再重复扣除导入日前的历史订单。
- 从导入时间之后继续累计库存变化。

### 完成标准

- [ ] 页面明确选择“期初库存”或“当日盘点”
- [ ] 导入日期必须填写
- [ ] 导入前展示预览
- [ ] 导入后生成批次编号
- [ ] 可按批次回滚

## 1.5 库存重算和差异检查

建立只读检查 SQL 或后台工具，至少支持：

- 按员工重算理论库存
- 按商品查看库存来源
- 比较理论库存与 `van_stocks`
- 找出不一致条码
- 找出没有流水但有库存的记录
- 找出有流水但没有库存记录的商品

### 完成标准

- [ ] 单员工重算通过
- [ ] 全员差异检查通过
- [ ] 差异可追溯到来源
- [ ] 重算前自动备份

## 1.6 回滚方案

每次改造前必须：

- 备份 `van_stocks`
- 备份相关订单
- 备份相关流水
- 保留旧 RPC 一段时间
- 支持切换回旧逻辑

# 第四阶段：登录、权限与 RLS

## 目标

系统不再只依赖员工编号或前端页面控制权限，数据库必须保证不同角色只能访问自己有权访问的数据。

## 4.1 角色设计

建议初始角色：

- `admin`
- `supervisor`
- `salesperson`
- `stock_manager`

### 权限建议

#### 业务员

- 查看自己的门店
- 查看自己的订单
- 查看自己的库存
- 创建和修改允许范围内的订单

#### 主管

- 查看所属团队
- 查看团队订单和库存
- 根据规则审核或处理订单

#### 库存管理员

- 导入库存
- 调整库存
- 查看库存流水
- 不一定拥有商品和员工管理权限

#### 管理员

- 全部管理权限
- 用户和角色管理
- 数据导出
- 恢复和审计

## 4.2 账号与员工绑定

建议新增：

```sql
user_profiles
```

字段建议：

- `user_id`
- `employee_code`
- `role`
- `team_id`
- `is_active`
- `created_at`

### 完成标准

- [ ] Supabase Auth 登录完成
- [ ] 用户与员工绑定
- [ ] 停用账号立即失效
- [ ] 页面不再信任 URL 中的员工编号

## 4.3 RLS 策略

实施顺序：

1. 先搭建登录和用户绑定。

5. 测试管理员、主管、业务员、库存管理员。

### 重点表

- [ ] 业务员不能伪造其他员工编号写入
- [ ] 主管只能查看所属团队
- [ ] 库存管理员只能执行授权操作

## 4.4 权限测试

必须验证：

- [ ] 前端隐藏按钮不是唯一权限控制
- [ ] 直接调用 Supabase API 也无法越权

- [ ] 导出接口受权限限制

- RLS 必须分表启用。

# 第五阶段：业务模块拆分

## 目标

减少大文件、全局变量和补丁覆盖，使订单、库存、售后、拼盒逻辑可以独立理解、测试和复用，为后续 React 重构打基础。

## 5.1 推荐目录结构

```text
src/
  domain/
    orderCalculation.js
    stockCalculation.js
    afterSalesRules.js
    mixBoxRules.js
    effectiveDateRules.js

  services/
    orderService.js
    stockService.js
    importService.js
    productService.js
    employeeService.js

  repositories/
    orderRepository.js
    stockRepository.js
    productRepository.js
    employeeRepository.js
```

## 5.2 拆分顺序

### 第一步：抽纯计算函数

优先抽离：

- 整件换算
- 散数换算
- 拼盒换算
- 售后净库存计算
- 7 月起算判断
- 订单金额计算

这些函数不直接访问页面和数据库，最容易测试。

### 第二步：抽数据库访问

把以下操作集中到 repository：

- 查询订单
- 查询库存
- 查询商品
- 提交订单 RPC
- 删除订单 RPC
- 导入库存 RPC

### 第三步：抽业务服务

页面只调用：

```js
await orderService.submitOrder(order)
await orderService.deleteOrder(orderNo)
await stockService.importOpeningStock(rows)
```

### 第四步：逐步移除补丁式覆盖

重点整理：

- `store-after-sales.js`
- `store-mix-box-edit-fix.js`
- `store-order-detail-fix.js`
- `store-stock-effective-date-fix.js`

将功能合并到正式模块，不长期保留多个 fix 文件相互覆盖。

### 完成标准

- [ ] 核心计算函数可单独测试
- [ ] 页面不直接计算最终库存
- [ ] 页面不直接拼接复杂数据库操作
- [ ] 补丁文件数量逐步减少
- [ ] 新开发者能按模块定位功能

# 每次实施的固定流程

以后每一项升级都按以下流程执行：

1. 阅读本文档和最新交接文档。
2. 明确本次只处理一个小范围目标。
3. 查询当前数据库结构和函数。
4. 查询相关 GitHub 文件。
5. 备份受影响数据。
6. 写测试或至少写可重复验证 SQL。
7. 在单员工或测试数据范围实施。
8. 核对订单数、明细数、库存数和流水。
9. 记录回滚 SQL。
10. 更新本文档状态。
11. 扩大到全员前再次确认。

# 禁止事项

- 不在没有备份时批量清表。
- 不使用 `TRUNCATE ... CASCADE` 处理生产业务表。
- 不让前端和数据库同时恢复库存。
- 不用前端传入的员工编号作为唯一权限依据。
- 不直接一次性开启所有表 RLS。
- 不在没有测试时全量 React 重写。
- 不把库存导入同时解释成“期初库存”和“当天盘点”。
- 不在来源不明确时直接修改历史订单库存。
